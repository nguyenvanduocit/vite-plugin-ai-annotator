# vite-plugin-ai-annotator

AI-powered browser element annotator for Vite. Users select UI elements in the browser, add feedback comments, and AI tools (Claude Code, Cursor, VS Code) receive structured element data via REST API and auto-setup skills.

## Commands

```bash
bun run build                 # All bundles: server + annotator + vite-plugin + types
bun run build:server          # CJS bundle -> dist/index.cjs (Node 18+)
bun run build:annotator       # IIFE bundle -> dist/annotator-toolbar.js (browser ES2020)
bun run build:vite-plugin     # ESM bundle -> dist/vite-plugin.js (Node, externals: vite)
bun run build:types           # Declarations via two tsconfigs (main + vite-plugin)
bun run typecheck             # tsc --noEmit (uses tsconfig.json, excludes vite-plugin.ts and rpc/define.ts)
bun src/index.ts              # Run server directly from source
bun link && bun link vite-plugin-ai-annotator  # Local dev linking (run build first)
```

## Publish

CI publishes to npm via OIDC Trusted Publishers — no `NPM_TOKEN` secret.

**Auto-release on push to master (default):**

The `Release` workflow runs on every push to `master`. `semantic-release` analyzes conventional-commit messages since the last tag, picks the bump (`feat:` → minor, `BREAKING CHANGE` → major, anything else conventional → patch), commits + tags + pushes, generates the GitHub Release, then a follow-up workflow step runs `npm publish --provenance --access public` in the workflow shell. The publish runs in the workflow shell deliberately — running it via `@semantic-release/exec` subprocess 404'd against the OIDC trusted-publisher endpoint, identical setup in a direct shell step works.

**Manual override:**

```bash
gh workflow run release.yml             # re-trigger semantic-release on current master
```

Use this when the auto-release run fails midway (e.g., commit + tag pushed but npm publish blew up). semantic-release sees the existing tag as baseline and resumes.

**Manual local publish (last-resort fallback):**

```bash
npm version patch -m "chore(release): %s"
git push --follow-tags
```

`NPM Publish` workflow triggers on the `v*` tag and publishes with provenance. Requires 2FA OTP if you publish from a workstation directly.

### ⚠ Commit-message gotcha: literal "BREAKING CHANGE"

The `conventional-commits` parser detects `BREAKING CHANGE` in a commit body even without the strict `BREAKING CHANGE: <description>` footer format — `BREAKING CHANGE -> major` (with a dash) is enough to trigger a major bump. Never write the literal phrase "BREAKING CHANGE" in a commit message body unless you actually intend to declare a breaking change. To talk about it, paraphrase ("breaking-change marker", "the BC keyword", etc.).

This bit us once already: the body of a `ci:` commit explaining release rules contained the literal text and shipped a `3.0.0` major bump.

## Architecture

```
src/
  index.ts               CLI entry. Starts the server
  ws-server.ts           Express + Socket.IO server. Two access layers:
                           - REST API at /api/* for any HTTP client
                           - Socket.IO for browser toolbar communication
  vite-plugin.ts         Vite plugin. Spawns server as child process, injects toolbar script
                           into HTML (SPA via transformIndexHtml, SSR via middleware interception)
  auto-setup-skills.ts   Writes AI tool skill/instruction files with server address:
                           - .claude/skills/ai-annotator/SKILL.md (Claude Code)
                           - .cursor/rules/ai-annotator.mdc (Cursor, alwaysApply)
                           - .windsurf/rules/ai-annotator.md (Windsurf, always_on)
                           - AGENTS.md (Codex, marker-delimited)
                           - .github/instructions/ai-annotator.instructions.md (Copilot)
                           - .clinerules/ai-annotator.md (Cline)
  annotator-toolbar.ts   Lit web component (<annotator-toolbar>). Shadow DOM, cyberpunk UI
  annotator/
    selection.ts         Element selection manager (badges, overlays, text wrapping)
    inspection.ts        Hover inspection mode (mouse tracking, highlight overlays)
    detectors.ts         Framework detection (Vue, React, Angular, Svelte)
    constants.ts         Z-index hierarchy, color palette, console limits
  rpc/
    define.ts            RPC interface definitions (ServerFunctions, ClientFunctions)
    *.generated.ts       Auto-generated by socketrpc-gen from define.ts
  utils/
    logger.ts            Verbose-gated logger factory
    screenshot.ts        Saves base64 webp to $TMPDIR/ai-annotator-screenshots/
    xpath.ts             XPath utilities
```

## Build System

Three esbuild bundles via `scripts/build-*.js` (CJS scripts, not ESM):

| Bundle | Entry | Format | Platform | Minified | External |
|--------|-------|--------|----------|----------|----------|
| Server | src/index.ts | CJS | node18 | yes | cors |
| Annotator | src/annotator-toolbar.ts | IIFE | browser es2020 | no | none |
| Vite Plugin | src/vite-plugin.ts | ESM | node18 | no | vite, child_process, path, fs, url |

Type generation runs twice: `tsconfig.json` (main, module=commonjs, excludes `vite-plugin.ts` and `rpc/define.ts`) then `tsconfig.vite-plugin.json` (module=ESNext, moduleResolution=bundler, only `vite-plugin.ts`).

## RPC System

`src/rpc/define.ts` is the source of truth:
- **ServerFunctions**: `getSessions`, `ping` - called by browser clients
- **ClientFunctions**: `getPageContext`, `getSelectedElements`, `triggerSelection`, `captureScreenshot`, `clearSelection`, `injectCSS`, `injectJS`, `getConsole`, `ping` - called on browser

Generated files (`*.generated.ts`) must not be edited manually. Regenerate with `socketrpc-gen` after changing `define.ts`.

## REST API

All endpoints under `/api/`. Session ID required in path (use `GET /api/sessions` to discover):

| Method | Endpoint | Body/Query | Description |
|--------|----------|------------|-------------|
| `GET` | `/api/sessions` | — | List connected browser sessions |
| `GET` | `/api/sessions/:id/page-context` | — | Page URL, title, selection count |
| `POST` | `/api/sessions/:id/select` | `{mode?, selector?, selectorType?}` | Trigger feedback selection |
| `GET` | `/api/sessions/:id/feedback` | `?fields=xpath,attributes,styles,children` | Get selected feedback items |
| `DELETE` | `/api/sessions/:id/feedback` | — | Clear all selections |
| `POST` | `/api/sessions/:id/screenshot` | `{type?, selector?, quality?}` | Capture screenshot, returns file path |
| `POST` | `/api/sessions/:id/inject-css` | `{css}` | Inject CSS into page |
| `POST` | `/api/sessions/:id/inject-js` | `{code}` | Execute JS in page context |
| `GET` | `/api/sessions/:id/console` | `?clear=true` | Get captured console logs |

## Auto-inject Feedback via Hook

Add a `UserPromptSubmit` hook to `.claude/settings.json` so every user message automatically fetches current feedback, injects it into context, and clears it:

```json
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'PORT=${AI_ANNOTATOR_PORT:-7318}; SESSIONS=$(curl -sf http://localhost:$PORT/api/sessions); [ -z \"$SESSIONS\" ] || [ \"$SESSIONS\" = \"[]\" ] && exit 0; echo \"$SESSIONS\" | python3 -c \"import sys,json; [print(s[\\\"id\\\"])for s in json.load(sys.stdin)]\" | while read -r SID; do FB=$(curl -sf http://localhost:$PORT/api/sessions/$SID/feedback?fields=xpath,attributes); if [ -n \"$FB\" ] && [ \"$FB\" != \"[]\" ]; then PAGE=$(curl -sf http://localhost:$PORT/api/sessions/$SID/page-context); echo \"[AI Annotator Feedback] Session: $SID\"; echo \"Page: $PAGE\"; echo \"Feedback: $FB\"; curl -sf -X DELETE http://localhost:$PORT/api/sessions/$SID/feedback > /dev/null; fi; done'"
          }
        ]
      }
    ]
  }
}
```

How it works:
1. On every user message, the hook queries `GET /api/sessions`
2. For each session with feedback, it fetches `GET /api/sessions/:id/feedback` and `page-context`
3. Output is injected as additional context into the prompt
4. Feedback is cleared via `DELETE /api/sessions/:id/feedback`
5. If no sessions or no feedback, the hook exits silently (exit 0)

Set `AI_ANNOTATOR_PORT` env var if using a non-default port.

## Gotchas

- **Two tsconfigs**: `tsconfig.json` uses `module: commonjs` (server code), `tsconfig.vite-plugin.json` uses `module: ESNext` (vite plugin needs ESM). Both needed for type generation
- **tsconfig.json excludes**: `src/rpc/define.ts` and `src/vite-plugin.ts` are excluded from main tsconfig
- **Vite plugin is dev-only**: `apply: 'serve'` means it does nothing during `vite build`
- **Source location injection**: `data-source-loc` attributes injected into HTML elements via MagicString regex transform (controlled by `injectSourceLoc` option, default true)
- **SSR support**: Vite plugin intercepts HTML responses via middleware (patches `res.write`/`res.end`) for frameworks like Nuxt
- **Server detection**: Plugin checks `/health` endpoint before spawning to avoid duplicate servers
- **Screenshots saved to tmpdir**: `$TMPDIR/ai-annotator-screenshots/screenshot-{timestamp}.webp`
- **Console capture**: Browser toolbar monkey-patches `console.*` methods, buffers max 1000 entries (trims to 500)
- **Default port**: 7318. Configurable via `--port`, `INSPECTOR_PORT`, or `PORT` env vars
- **Socket.IO max buffer**: 50MB (`maxHttpBufferSize`) for large screenshots
- **Build scripts are CJS**: `scripts/build-*.js` use `require()`, not ESM imports
